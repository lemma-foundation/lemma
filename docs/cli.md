# CLI

The public CLI is a thin reference client. It gets miners and validators through setup, status checks, local mining smoke tests, and validation. It is not the competitive mining engine.

## Public Commands

```bash
uv run lemma setup
uv run lemma status
uv run lemma mine --once --prover-command "python prover.py" --output submission.json
uv run lemma validate --once --submission-spool submission-spool --no-set-weights
```

`setup` writes local task, corpus, active-window, wallet, unearned-allocation, and optional prover settings. `status` shows the active registry, verifier settings, wallet names, and prover command.

Local `LEMMA_TASK_REGISTRY_URL` or `file://` task-registry inputs must resolve to regular files; HTTP(S) registry fetching remains unchanged.

`mine` runs one reference proof-search iteration. It sends a task JSON object to the configured prover command and expects a JSON proof response on stdout. Serious miners should build their own agents, workers, models, schedulers, or direct protocol clients.

`validate` loads active tasks, rejects malformed submissions, checks proofs, scores accepted work, writes score events, and appends `validator-runs.jsonl`. A submission spool is a top-level directory of pending `.json` or `.jsonl` files; after a successful validator pass, consumed files move to `processed/`.

Ingredient JSONL component files are sorted by each row's public component ID before component hashing.

`ingredients verify-bundle --generation-receipt-envelope` may be repeated to add external regular-file envelopes to the bundle's embedded envelope for quorum and signature verification.

Ingredient repository `--root` inputs must be concrete directories before root-relative components, reports, recipe artifacts, selection receipts, or gate receipt evidence are read.
Ingredient repository output roots for extraction and compatibility scaffolding must be absent or concrete directories before public root-relative artifacts are written.
Ingredient repository writers reject symlinked or non-regular root-relative artifact paths before writing or preserving public artifacts.

Soundness-template import lines must use the same duplicate-free, sorted, Mathlib-only public import envelope as generated ingredient tasks.

Statement-gate receipts explicitly bind the selected parameter map, its canonical hash, and the generated theorem-type hash, so validators can replay the realized challenge context without trusting task metadata alone.

`ingredients verify-task` also rejects task envelope drift outside the Lean target hash: source-license drift, non-restricted submission policy, verifier or toolchain drift, lifecycle windows, active queue/frontier/difficulty drift, non-Mathlib imports, nonempty titles, and unknown task metadata.

Ingredient manifests, raw ingredient rows, selection receipts, generation receipts, gate receipts, generation-receipt envelopes, triviality-gate probe hashes, novelty-gate detail hashes, and task artifact manifest hashes reject the all-zero placeholder before public replay; public Mathlib commits, ingredient repository commits, gate runner/check tokens, and envelope signer/signature metadata must also be non-placeholder, and bundle artifact refs must carry non-placeholder file hashes.

Selection receipt selected selector and recipe IDs must be canonical public labels, and selected definition, fact, and bridge ID arrays must be duplicate-free before public receipt hashing.
Quality reports that advertise nonzero `estimated_theorem_space_size` must mark `reserve_selector_health.ready=true`.

Statement-gate and shortcut-gate receipt details repeat the selected selector ID from the public selection receipt, so validators can see which hash-ordered selector path produced the task without trusting adjacent task metadata.
Task artifact manifests carry the active target hash, theorem statement hash, selected selector/recipe IDs, selected-parameter hash, theorem-type hash, novelty-family hash, corpus snapshot hash, ingredient repository commit, Mathlib commit, and recipe bundle hash from the public task, manifest, selection receipt, and generation receipt. `tasks build-ingredient-task`, `ingredients verify-task`, and `ingredients verify-bundle` JSON summaries also expose the selected selector/recipe IDs, and task build/bundle summaries echo the manifest-bound provenance pins.

Generation receipts, gate receipts, and task artifact manifests require active task IDs under the `lemma.ingredient.*` namespace.

Ingredient fact IDs form one public namespace across `facts.jsonl`, `source_theorems.jsonl`, and `source_lemmas.jsonl`; root inspection, selection replay, and shortcut receipt recomputation reject cross-catalog duplicates.

Fact rows marked with `metadata.usable_as_source_fact=false` remain public catalog facts for inspection and exact-match shortcut scanning, but deterministic raw-ingredient selection skips them.

Fact metadata fields `statement_family`, `topic`, and `subtopic` must be canonical public labels, not free-form text.

Fact dependency metadata `direct_dependency_count` and `dependency_depth` must be exact nonnegative integers when present.

Ingredient novelty cache JSONL rows are canonical sorted objects containing either `statement_hash` or `novelty_family_hash`. When a novelty cache is supplied, challenge building and bundle verification bind both theorem-type novelty and selected-family novelty.

Definition rows with `metadata.allowed_recipes` are eligible only for those listed recipes during deterministic selection.

Definition metadata `simp_risk`, when present, must be one of `low`, `medium`, or `high`.

Selector `ingredient_filters.max_simp_risk`, when present, excludes recipes whose required definitions exceed that public risk cap; definitions without `simp_risk` are treated as `high`. Selector `ingredient_filters.min_dependency_depth`, when present, requires selected facts to carry public `metadata.dependency_depth` greater than or equal to that minimum.

Bridge rows are usable for a safe recipe only when both bridge domains appear in that recipe's public domain list.

Compatibility edges may only declare allowed domains from their recipe domain list and may only reference bridges that are safe for that recipe and whose bridge domains are also declared by the recipe.

Task metadata carries one canonical generation receipt hash. `ingredients verify-task --generation-receipt-envelope-quorum` and `ingredients verify-task --verify-envelope-signatures` require `--generation-receipt-envelope` inputs; without envelopes, the command fails closed instead of reporting reconstructed metadata as verified envelope evidence.

Recipe rows with versions other than `1`, nonempty `preconditions`, or nonzero `difficulty_delta` values fail public root inspection until those fields are executable and replayable by the validator path.
Recipe `soundness_template` paths must stay under `soundness_templates/` with a canonical public-label file stem. The directory may contain only direct `.lean` template files, and every template file in that directory must be declared by a recipe.

## Protocol Contract

The stable surface is the protocol output, not the CLI implementation. Miners can use any infrastructure that produces valid task-bound proof submissions. Validators configure their environment and run `lemma validate`; lower-level diagnostics stay out of the normal path.

Task, submission, verification-result, score-event, and corpus-row shapes live under `spec/`. Proof Atlas and registry operator flows are documented separately in [Proof Atlas](proof-atlas.md) and [Operator Registry Flow](operator-registry-flow.md).
